Architecture in one sentence

The browser tools and the native Visualizer Studio generate or manage scenes, the native Visualizer validates and stores the newest accepted scene, and the OpenGL renderer draws that stored scene every frame.

Human-readable architecture map

Control Center

Browser orchestration, docs links, API lab, and managed-process launching.

Activity Monitor

Browser log viewer over the shared JSON-lines activity journal.

Scene Studio

Browser preset authoring and live browser-side scene control.

Visualizer Studio

Unified native desktop source selection, preview, validation, apply, and live streaming.

HTTP API

Validation, submission, example scenes, runtime limits, and runtime logs.

Scene store

The thread-safe source of truth for the latest accepted scene.

Renderer

The OpenGL loop that uploads and draws the current scene on the GPU.

How a scene reaches the screen

  1. A tool such as Visualizer Studio or Scene Studio generates a scene JSON document.
  2. The tool optionally validates it through POST /api/v1/scene/validate.
  3. The tool submits it through POST /api/v1/scene.
  4. The HTTP API parses the JSON and turns it into typed scene data.
  5. The scene store replaces the current scene snapshot.
  6. The renderer sees the new scene version and uploads it to GPU buffers.
  7. The GPU draws the frame.

How shared logging works now

The native Visualizer, the browser Control Center, and Visualizer Studio all append structured JSON-lines events to the same shared activity journal. The Activity Monitor reads that journal in the browser and lets you sort and filter it by time, source app, level, and text.

Why bar walls are no longer a separate renderer

A bar wall is just a specialized 3D scene. Treating it as a separate renderer created extra launchers, extra docs paths, extra operator tools, and extra mental overhead. The current architecture treats bar walls as one more scene family inside the same renderer and API, which is simpler to learn and simpler to maintain.
