What this page is about

This page is intentionally about the runtime architecture of Halcyn: the pieces that are alive while the app is being used. It focuses on who owns the scene data, how requests move through the process, and where rendering actually happens.

This page deliberately leaves out CI, packaging, branch rules, and release mechanics. Those matter for engineering quality, but they are not part of the runtime path that turns requests into pixels.

Runtime map

Stage 1

Entry points

These are the ways a person or tool asks Halcyn to do something. None of these entry points draw directly; they all feed into a smaller number of canonical runtime services.

Desktop startup

PowerShell launch scripts and command-line arguments choose the startup sample, the HTTP port, the window size, and optional scene files.

Browser Control Center

The browser dashboard helps with launching, testing, docs access, log viewing, and request orchestration before or during a desktop app session.

Browser Scene Studio

The scene authoring UI sends control changes that become scene updates or server-driven live-session output.

Desktop Render Control Panel

The native operator console offers local audio-device selection, a desktop pointer pad, instant 2D and 3D switching, and its own live-stream controls.

Spectrograph Suite

A dedicated renderer executable plus a native desktop operator panel transform generic JSON into an adaptive 3D bar wall before submitting ordinary scene JSON into the shared renderer pipeline.

Direct HTTP callers

PowerShell scripts, tests, curl, and other tools can submit or validate scene JSON without going through a browser at all.

Stage 2

Acceptance pipeline

Once a request reaches the C++ application, the data follows one deliberate path so parsing, validation, and rendering stay clearly separated.

src/http_api

The embedded HTTP server receives requests, enforces payload limits, and turns success or failure into readable JSON responses.

src/scene_description/SceneJsonCodec

The codec parses raw JSON into typed scene objects and can serialize stored snapshots back into JSON for diagnostics.

ValidateSceneDocument

Semantic validation checks things the JSON parser alone cannot prove, such as primitive alignment, index bounds, and sane 3D camera settings.

BuildRenderScene

Both 2D and 3D scenes are normalized into one render-friendly shape so the GPU upload path can stay simple, even when 3D scenes also carry presentation options such as shader style and anti-aliasing.

Stage 3

Authoritative shared runtime state

After validation succeeds, the new scene becomes shared application state. This is the hand-off point between "request handling" and "ongoing rendering".

SceneStore

Owns the current immutable scene snapshot, including version number, source label, and activation time. This is the renderer's source of truth.

RuntimeLog

Keeps the shared log stream used by the terminal, the embedded API, and the browser tooling. This is the observability spine of the app.

desktop_app::Application

Wires the codec, scene store, log, API server, and renderer together and owns their startup/shutdown order.

Stage 4

Frame production

Rendering begins only after a valid snapshot already exists. The renderer never needs to parse JSON or reason about partially updated scene state.

Renderer

Watches the snapshot version, uploads new buffers only when the scene changes, and drives the frame loop on the main thread.

ShaderProgram

Wraps shader compilation, linking, and uniform uploads so the render loop can focus on rendering decisions instead of low-level program management.

OpenGL + GPU window

The GPU transforms vertices, rasterizes primitives, blends colors, and fills the framebuffer that eventually appears in the visible app window.

Request-to-pixels sequence

  1. A person, script, Control Center action, Scene Studio control, or spectrograph control-panel action emits a request or startup instruction.
  2. The desktop app or embedded HTTP API receives that input and turns it into a scene loading or scene submission attempt. In the spectrograph workflow, the desktop panel first converts generic JSON into a full 3D bar scene.
  3. The JSON codec converts raw text into typed scene objects and preserves readable error information if the input is malformed.
  4. Semantic validation checks that the scene would make sense to draw, not just that the JSON has the right shape.
  5. The validated scene is converted into the normalized render format and published into the shared scene store as a brand-new immutable snapshot.
  6. The renderer notices the new snapshot version on a later frame and uploads the latest vertex and index data to GPU buffers.
  7. The renderer builds either an orthographic 2D matrix or a perspective 3D matrix, depending on the scene kind.
  8. OpenGL executes the draw call and the GPU fills the window with the new image.

Responsibility boundaries

src/desktop_app

  • Owns startup, shutdown, and command-line parsing.
  • Does not own HTTP transport details or OpenGL draw logic.

src/http_api

  • Owns request routing, response formatting, and transport-level checks.
  • Does not own the canonical scene model or GPU behavior.

src/scene_description

  • Owns the scene language, JSON codec, examples, and validation rules.
  • Does not own threading, sockets, or rendering loop state.

src/shared_runtime

  • Owns immutable scene snapshots and shared log history.
  • Does not own parsing rules or drawing commands.

src/opengl_renderer

  • Owns GLFW, glad, shader use, matrix building, buffers, and draw calls.
  • Does not own JSON parsing, request validation, or browser orchestration.

Operator tooling

  • Owns operator workflow, authoring UX, local device selection, and orchestration affordances.
  • Does not replace the scene store as the canonical source of truth.

Why there are multiple control surfaces

The Control Center, Scene Studio, and Desktop Render Control Panel serve different jobs. The Control Center is about orchestration: builds, tests, logs, docs, and API experiments. The Scene Studio is about browser-native live scene authoring. The desktop panel is about a native operator experience with local hardware awareness. Keeping them separate avoids one overloaded UI that tries to be every tool at once.
