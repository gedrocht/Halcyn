What this app is for

This desktop app solves one focused problem: it listens to a real local audio source, turns the recent activity into a readable JSON document, and sends that JSON into Signal Router or Bars Studio.

Think of Signal Router and Bars Studio as the places that decide how the visuals should behave, while Audio Sender decides what live audio data should feed those visuals.

Why the audio source lives in a separate app

Cleaner learning path

A beginner can study capture and rendering as two related problems instead of one huge window that tries to explain everything at once.

Safer boundaries

The apps talk through a tiny local bridge contract instead of reaching into each other's UI controls directly.

Better reuse later

The same capture logic can grow independently if you later want more sources, more analysis, or more renderer targets.

Two-app workflow

If you want the whole bar-wall audio path opened for you, run .\scripts\launch-bars-workbench.ps1. That helper opens the Visualizer, Signal Router, and this Audio Sender in separate windows.

  1. Start the bar-wall renderer with .\scripts\launch-visualizer.ps1 -Sample spectrograph -Port 8090, or simply use .\scripts\launch-bars-workbench.ps1.
  2. Start Signal Router with .\scripts\launch-signal-router.ps1 if it is not already open.
  3. Switch Signal Router to External feed if you want it to follow helper-app input directly.
  4. Start this app with .\scripts\launch-audio-sender.ps1.
  5. Choose Output sources if you want desktop loopback audio, or Input sources if you want a microphone-style source.
  6. Pick the quick bridge target for Signal Router or Bars Studio.
  7. Pick a device, click Start capture, and watch the volume bar move.
  8. Use Send once to test the bridge, or Start live for continuous updates.

What the app can control

Audio source flow

Switch between desktop output loopback and microphone-style input devices.

Device selection

Refresh the device list, choose the active device, and start or stop capture explicitly.

Volume monitor

Watch one simple live meter so you can immediately tell whether the chosen source is active.

Rolling history

Choose how many recent frames should be kept when the app builds the outgoing JSON document.

Bridge target

Choose the host, port, path, and source label for the local bridge delivery.

Live cadence

Set how frequently the generated JSON should be posted while live sending is running.

What the generated JSON contains

The output intentionally stays generic. That means Signal Router or Bars Studio can keep using its existing JSON flattening pipeline instead of learning a new binary or specialized audio protocol. 

{
  "sourceType": "audio-sender",
  "device": {
    "name": "Desktop speakers",
    "identifier": "speaker-1",
    "flow": "output"
  },
  "summary": {
    "available": true,
    "capturing": true,
    "currentLevel": 0.71,
    "currentBass": 0.62,
    "currentMid": 0.33,
    "currentTreble": 0.18,
    "averageLevel": 0.46,
    "peakLevel": 0.88
  },
  "audioFrames": [
    { "level": 0.12, "bass": 0.08, "mid": 0.04, "treble": 0.01 },
    { "level": 0.71, "bass": 0.62, "mid": 0.33, "treble": 0.18 }
  ]
}

If you open the JSON preview window, you can study the full document that will be sent across the local bridge.

How the bridge works

Signal Router and Bars Studio each expose a tiny local HTTP server at /external-data. This app posts JSON to that route. The receiving tool then treats that incoming text as its newest external JSON source and can use it during preview, validate, apply, or live-stream operations.

That design keeps the boundary simple: one app is responsible for capture, and the other remains responsible for translating generic JSON into the final scene or bar wall.

If you want to inspect successful deliveries, skipped unchanged sends, or bridge failures, open .\scripts\launch-activity-monitor.ps1.
