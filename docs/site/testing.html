Why the tests are layered instead of all living in one place

Halcyn now has native C++ rendering code, browser-side Python tooling, and a desktop Python control panel. The deterministic pieces are pushed into automated tests, while the genuinely hardware-dependent pieces still keep a smaller manual-check list.

Current automated coverage

Native scene pipeline

Valid 2D and 3D payloads are parsed, validated, stored, and normalized predictably.

Browser Control Center

Lint, type checks, coverage, and repository-contract tests keep the orchestration UI honest.

Desktop control panel

Unit tests cover audio service fallbacks, HTTP client behavior, controller logic, and the hidden Tk window flow.

How to run the tests

.\scripts\run-native-tests.ps1
.\scripts\run-native-tests.ps1 -Configuration Release
.\scripts\test-browser-control-center.ps1
.\scripts\measure-browser-control-center-coverage.ps1
.\scripts\test-desktop-render-control-panel.ps1
.\scripts\measure-desktop-render-control-panel-coverage.ps1

If you want the same consolidated pass CI expects locally, run .\scripts\run-all-quality-checks.ps1 -Configuration Debug.

Recommended manual checks

  • Post the 2D triangle sample and confirm the window updates.
  • Post the 3D tetrahedron sample and confirm depth ordering looks correct.
  • Post intentionally invalid JSON and confirm the API returns a 400 response with readable errors.
  • Resize the window and confirm the scene stays visible.
  • Launch the desktop render control panel and confirm it can preview, validate, and apply a scene.
  • If you have audio hardware available, confirm the desktop panel can enumerate a device and react to changing input levels.
