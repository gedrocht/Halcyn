Overview

Send JSON in. See 2D or 3D graphics on the GPU.

Halcyn is a C++20 application with an embedded HTTP API and an OpenGL renderer. One thread accepts JSON requests. Another thread-safe layer stores the latest validated scene. The main thread renders that scene in a real window at a 60 FPS target using the GPU.

What problem Halcyn solves

Many graphics demos focus only on rendering. Many API demos focus only on receiving JSON. Halcyn combines both: it gives you a small but complete system that accepts structured JSON scenes and turns them into a visible GPU-rendered result. That makes it a good teaching project for learning how application layers cooperate in a real codebase.

API layer

Receives JSON over HTTP, validates it, and returns readable error messages when a request is invalid.

Scene description layer

Defines what a valid 2D scene and a valid 3D scene actually are.

OpenGL renderer layer

Uploads vertices to OpenGL buffers and asks the GPU to draw the latest scene.

Important beginner idea: CPU vs GPU

The CPU still opens the window, accepts requests, parses JSON, validates data, and tells the GPU what to draw. The GPU is the part that transforms vertices, rasterizes primitives, blends colors, and fills the screen with pixels. That division of labor is exactly what modern graphics software is supposed to do.

If you are new to graphics, think of the CPU as the planner and the GPU as the specialist drawing team.

Project tour

Path Why it exists
src/scene_description Scene types, validation, JSON codec, and scene factory helpers.
src/shared_runtime Thread-safe storage for the latest scene snapshot and shared runtime logs.
src/http_api Embedded HTTP routes for health, scene retrieval, and scene submission.
src/opengl_renderer Window creation, shader setup, GPU uploads, and frame loop.
src/desktop_app Application startup, command-line parsing, and the glue that wires the runtime together.
tests Unit tests for parsing, validation, and scene store behavior.
scripts Easy entry points for building, running, testing, posting sample JSON, and serving docs.
desktop_render_control_panel A native Tk-based operator console for local audio-device control, 2D/3D switching, preview, validate, and live streaming.
desktop_spectrograph_control_panel Bars Studio, a native operator console that turns generic JSON into a statistically normalized 3D bar-wall scene.
desktop_spectrograph_audio_source_panel Audio Sender, a native helper app that captures live audio, builds readable JSON, and feeds another local desktop bridge.
desktop_multi_renderer_data_source_panel Signal Router, a native routing console that captures one data source and sends translated scenes to the scene renderer, the bar-wall renderer, or both.
desktop_shared_control_support Shared desktop-side audio, bridge, logging, and renderer-client support surfaces used by multiple native tools.
src/spectrograph_main.cpp A compatibility entry point that gives the shared engine a bar-wall-oriented default personality.
