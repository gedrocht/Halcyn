What it is

Scene Studio is the separate browser-facing scene authoring surface for Halcyn. It lives in its own top-level folder, but it still relies on the existing Control Center server for routing, live-session control, logging, and test coverage.

This page explains the tool, but the live Scene Studio UI runs locally. Launch it with the script below, or open /scene-studio/ from an already running local Control Center session. 
.\scripts\launch-browser-scene-studio.ps1

Signal inputs

Unix time

Provides a stable animated phase even when no hardware input is active.

Noise

Adds deterministic variation so repeated frames do not feel too rigid.

Pointer

Uses normalized mouse position and motion speed captured in the browser.

Microphone

Uses browser-side microphone analysis when the user explicitly enables it.

Manual drive

Lets the operator push the preset without depending on external input at all.

Why this stays in the same repo

The client and renderer share one scene contract. Keeping the client in this repository means the contract, docs, CI, and release rules stay synchronized. The UI is still physically separated under browser_scene_studio/, so it can evolve independently without dragging the rest of the Control Center around.

How live streaming works now

The browser does not need to send a full JSON scene every time the pointer or audio input changes. Instead, it sends lighter control updates to the Control Center. The Control Center keeps a persistent live session, regenerates scenes server-side on a fixed cadence, and streams those scenes into the live Halcyn API.

Session status now travels back to the browser over a server-sent event stream, so the client does not need to keep polling the Control Center every few hundred milliseconds just to learn whether anything changed.

Why it is still build-free HTML instead of React

The main latency improvement came from moving the streaming loop to the server side, not from changing frontend frameworks. Keeping the client as a static HTML, CSS, and JavaScript app preserves the repo's existing quality gates on machines without a Node toolchain. If the UI grows more state-heavy later, the backend contract is now stable enough to support a React migration without changing the renderer-facing design.

How it differs from the native desktop panel

Scene Studio is best when you want a browser-first creative surface and do not mind browser permissions for microphone input. The Desktop Render Control Panel is best when you want local audio-device selection, a native pointer pad, and a second window that can sit beside the renderer without relying on browser tabs.

Control Center routes behind the client

Scene Studio uses the catalog, preview, one-shot apply, and live-session routes under /api/scene-studio/.... The live-session endpoints are what make the browser feel more responsive during continuous manipulation.

The session stream route is the lean part of that loop: it lets the Control Center push session updates when something actually changes instead of making the browser ask over and over again.
