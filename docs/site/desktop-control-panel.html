What it is

The Desktop Render Control Panel is a native Tkinter application that talks to the same Halcyn HTTP API as the browser tools, but presents the controls in a desktop window that can live beside the renderer itself. 

.\scripts\launch-desktop-render-control-panel.ps1
GitHub Pages can document this tool, but the real control panel runs locally on your machine because it needs direct access to your desktop environment and optional audio hardware.

What it can do

Instant 2D and 3D switching

One native panel can move between desktop-only 2D presets and the shared 3D preset catalog immediately.

Local audio-device selection

Unlike a browser-only surface, the desktop panel can default to output sources for desktop loopback capture, switch over to microphone inputs when needed, and explain which optional package is responsible for each path: soundcard for output loopback and sounddevice for microphones.

Preview, validate, apply, and save

You can open the generated JSON in its own window when you need it, validate it against the live API, apply it immediately, or save the whole setup to a reusable JSON file.

Continuous live streaming

The panel can keep submitting scenes on a controlled cadence while you adjust colors, density, motion, and signals.

Main areas in the window

Renderer connection

Choose the API host, port, and live cadence, then see the exact cadence value in milliseconds beside the slider before you run health, validate, apply, start, or stop actions.

Scene controls

Choose scene type and preset with stay-selected buttons, then tune geometry density, motion speed, gain, manual drive, and live color swatches.

Audio input

The roomier diagnostics column now holds audio controls, including a toggle between output sources and input sources plus one easy-to-read live volume meter. Output sources are selected by default, and the panel can explain when soundcard is needed for loopback capture or when sounddevice is needed for microphones.

Pointer pad, settings, and preview

Drive pointer-based motion on a larger pad, revert to preset defaults, save or load settings, and open the exact JSON scene in a dedicated window with a Copy JSON button instead of stretching the main panel.

Dark mode and control feel

The polished desktop panel now ships in a dark-mode-first style so it feels more at home beside the renderer window. Integer sliders snap to whole values, floating-point sliders snap to practical increments, the currently selected 2D or 3D mode stays visibly pressed with high-contrast text, and the heavier pointer/audio controls now live in the right-hand column so the scene controls stay reachable on 1080p displays.

How it fits with the browser tools

The browser Control Center is still the orchestration dashboard for builds, tests, logs, docs, and API experiments. The browser Scene Studio is still a browser-native creative surface for live authoring. The Desktop Render Control Panel is the native operator console for the situations where local hardware awareness and a desktop companion window matter more than staying in a browser tab.

If your goal is not general scene authoring but generic-data-to-bar-wall conversion, the dedicated Spectrograph Suite is the more specialized tool.

Testing and quality gates

The desktop panel is treated like the rest of the repo's maintained code, not like an experimental extra. It has linting, type checks, unit tests, and a 90% coverage gate. 

.\scripts\lint-desktop-render-control-panel.ps1
.\scripts\typecheck-desktop-render-control-panel.ps1
.\scripts\test-desktop-render-control-panel.ps1
.\scripts\measure-desktop-render-control-panel-coverage.ps1

Optional audio dependency

Real device capture depends on two optional Python packages. The desktop panel uses sounddevice for microphone and line-input capture, and soundcard for desktop output-loopback capture. If either package is missing, the panel still opens and the rest of the controls still work; only that capture path is disabled. On Windows, the panel still tries to list available input devices through the built-in waveIn API so the operator can see what hardware exists before installing sounddevice. 

python -m pip install sounddevice soundcard
